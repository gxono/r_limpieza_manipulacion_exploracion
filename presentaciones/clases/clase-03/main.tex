	\begin{frame}
		Este eje proporciona las herramientas necesarias para trabajar con los formatos de archivo más comunes en el ámbito profesional y académico. 
		
		Abordaremos la lectura de archivos de Excel (formato propietario ampliamente utilizado), archivos de texto delimitado (CSV, TSV) y otros formatos estructurados. 
		
		Veremos c\'omo gestionar proyectos de RStudio mediante archivos .Rproj, estableciendo flujos de trabajo organizados con rutas relativas que favorecen la reproducibilidad. 
		
		Revisaremos algunas buenas prácticas para la inspección inicial de los datos importados, verificando tipos de columnas, dimensiones y detección temprana de problemas de codificación o formato.
	\end{frame}
	
	
	
	
	
	\begin{frame}{Gesti\'on de proyectos en R con RStudio}
		Uno de los problemas más comunes al comenzar a programar es la gestión de los archivos. ¿Dónde están mis datos? ¿Dónde se guardan mis gráficos? \shaky{¿Por qué mi código no corre en la computadora de mi compañerooo?} La respuesta suele estar en el manejo del "directorio de trabajo" y los "proyectos".
	\end{frame}



	\begin{frame}[fragile]{?`Qué es y cómo configurar el directorio de trabajo?}
		El directorio de trabajo (Working Directory) es la carpeta donde R \quotation{mira} por defecto. Si le ped\'is leer un archivo \texttt{datos.csv}, R lo buscará en esa carpeta. Si guard\'as un gráfico, se guardará ahí.
		
		Pod\'es saber en qué carpeta estás \quotation{parado} con el comando
\begin{lstlisting}[style=RScript]
getwd() # "get working directory"
\end{lstlisting}		
		Y pod\'es cambiar esa carpeta con
\begin{lstlisting}[style=RScript]
setwd("C:/Usuarios/MiUsuario/Documentos/MiCursoR") # "set working directory"	
\end{lstlisting}
	\end{frame}
	
	
	
	\begin{frame}[fragile]
		Usar \rscript{setwd()} con rutas absolutas (como la de arriba) es una mala práctica porque hace que tu código sea frágil. Si mov\'es la carpeta, le cambi\'as el nombre, o le pas\'as el script a otra persona con otro usuario, el código fallará porque la ruta \rscript{"C:/Usuarios/MiUsuario..."} no existirá.
		
		Para evitar esto, tenemos los proyectos.
	\end{frame}
	
	
	
	\begin{frame}[fragile]{Proyectos}
		Para solucionar esto, RStudio introdujo los R Projects (\verb`.Rproj`). Un proyecto es un archivo que define la raíz de tu trabajo. Cuando abres un proyecto, RStudio automáticamente configura el directorio de trabajo en la carpeta donde está ese archivo.
		
		\titleitem{Pasos para crear un proyecto}
		\begin{enumerate}
			\item And\'a a \verb`File > New Project`.
			\item Eleg\'i \verb`New Directory` para una carpeta nueva o \verb`Existing Directory` si ya ten\'es la carpeta preparada.
			\item Eleg\'i \verb`New Project`.
			\item Dale un nombre a la carpeta y eleg\'i dónde guardarla.
		\end{enumerate}
		
		
		Al abrir este proyecto, \rscript{getwd()} siempre apuntará a la carpeta del proyecto. Esto hace que tu trabajo sea portable.
	\end{frame}
	
	
	
	
	\begin{frame}[fragile]{Rutas relativas}
		Exacto, tener el proyecto no es suficiente si seguimos siendo desordenados. Para aprovechar la portabilidad, debemos usar rutas relativas y una estructura de carpetas consistente.
		
		\titleitem{Estructura recomendada}
		Dentro de tu carpeta de proyecto, crea subcarpetas:
		\begin{itemize}
			\item \verb`datos/`: Ac\'a guard\'a tus excels, csv, etc. 
			\item \verb`scripts/`: Ac\'a guard\'a tus archivos \verb`.R` o \verb`.qmd`. 
			\item \verb`resultados/`: Ac\'a guard\'a los gráficos o tablas que exportes.
		\end{itemize}
		 
		
		\titleitem{Rutas Relativas}
		En lugar de decirle a R dónde está el archivo desde el disco C (\verb`"C:/..."`), le dec\'is dónde está desde la carpeta del proyecto.
		\begin{itemize}
			\item \textcolor{red}{\textbf{MALA PRÁCTICA}}: \verb`"C:/Usuarios/Juan/ProyectoR/datos/tabla.csv"`
			\item \textcolor{green!60!black}{\textbf{BUENA PRÁCTICA}}: \verb`"datos/tabla.csv"`
		\end{itemize}		
		
		Si usas rutas relativas, puedes mover toda la carpeta del proyecto a cualquier computadora y seguirá funcionando \emoji{wink}.
	\end{frame}
	
	
	
	\begin{frame}[fragile]{Carga de datos}
		En general, no vamos a trabajar con un entorno cerrado. Siempre necesitaremos datos externos. Los m\'as frecuentes, y que no son datos que vayamos a analizar, son los paquetes.
		
		\titleitem{Los paquetes}
		Los paquetes son piezas de c\'odigo empaquetadas que ofrecen funcionalidades  que el software de base no implementa o que son engorrosas de escribir. Por ejemplo, la funci\'on \rscript{lag()} (que me imagino que ya buscaste para qu\'e sirve) no viene implementada por defecto en R, por lo que debe ser incorporada mediante un paquete.
	\end{frame}
	
	
	\begin{frame}[fragile]{Instalar un paquete}
		Para instalar un paquete en nuestro sistema ejecutamos \rscript{install.packages()} y le pasamos como cadena de texto (entre comillas) el nombre del paquete.
		
\begin{lstlisting}[style=RScript]
install.packages(c("pacman", "tidyverse", "janitor"))
\end{lstlisting}

	No es necesario que esta instrucci\'on vaya al script. Puede ejecutarse desde la consola.
	
	Ahora bien, ahora el/los paquetes estar\'an instalados en nuestro sistema. Es necesario cargarlos al proyecto. Para cargar un paquete usamos la instrucci\'on \rscript{library()} y le pasamos el nombre del paquete como variable (sin comillas).
\begin{lstlisting}[style=RScript]
library(tidyverse)
library(janitor)
\end{lstlisting}

Esto s\'i debe ir en el script.
	\end{frame}
	
	
	
	
	
	
	\begin{frame}[fragile]{pacman}
		La librer\'ia \rscript{pacman} es \'util si necesitamos cargar paquetes y no sabemos si est\'an instalados o no. En lugar de probar uno por uno, y luego cargar, podemos hacer algo as\'i
		
\begin{lstlisting}[style=RScript]
library(pacman)
pacman::p_load(fixest, modelsummary)
\end{lstlisting}

Esto chequea si existe \rscript{fixest} y \rscript{modelsummary}. Si no existen, los descarga. Luego, carga los paquetes.
	\end{frame}
	
	
	\begin{frame}[fragile]{?`Y por qu\'e ::?}
		Si dos paquetes distintos definen una funci\'on con el mismo nombre (por ejemplo, la librar\'ia est\'andar \rscript{stats} de R y \rscript{dplyr} de Posit tienen una versi\'on propia de \rscript{lag}) podemos elegir una u otra con ese prefijo:
\begin{lstlisting}[style=RScript]
stats::lag() #lag de R
dplyr::lag() #lag de dplyr
\end{lstlisting}
		
			Tambi\'en puede ser útil para saber qu\'e funciones, objetos o data frames trae la librer\'ia como se puede ver en la siguiente imagen con la \rscript{eph}.
		\begin{figure}
			\includegraphics[width=0.75\linewidth]{img/eph.png}
		\end{figure}
		
		Aunque para esto \'ultimo es mucho m\'as útil ejecutar algo como
\begin{lstlisting}[style=RScript]
data(`:ky:package` = "eph") #Intentalo :)
\end{lstlisting}
	\end{frame}

	
	\begin{frame}[fragile]{Carga de datos}
		R puede leer casi cualquier formato de datos. Veremos los más importantes. Para esto, utilizaremos principalmente el paquete \rscript{readr} (incluido en tidyverse) para texto plano, y paquetes específicos para otros formatos.
		
		Recuerda cargar \rscript{tidyverse} antes de empezar:
\begin{lstlisting}[style=RScript]
library(tidyverse)	
\end{lstlisting}			
	\end{frame}
	

	
	
	\begin{frame}[fragile]{Ficheros .xlsx}
Los archivos de Microsoft Excel son el estándar en muchas industrias. Para leerlos, usamos el paquete \rscript{readxl} (parte del ecosistema \rscript{tidyverse}, pero a veces se debe cargar explícitamente).
\begin{lstlisting}[style=RScript]
library(readxl)

# Leer la primera hoja del archivo
datos <- read_excel("datos/mi_excel.xlsx")

# Leer una hoja específica (por nombre o índice)
datos_ventas <- read_excel("datos/mi_excel.xlsx", sheet = "Ventas 2023")	
\end{lstlisting}

\rscript{read_excel()} detecta automáticamente el tipo de dato de cada columna, pero puedes forzarlo con el argumento \rscript{col_types} si es necesario.
	\end{frame}




\begin{frame}[fragile]{Ficheros .ods}
Los archivos Open Document Spreadsheet (.ods) son la alternativa libre a Excel (usados por LibreOffice, OpenOffice, Google Sheets). Para leerlos, necesitamos el paquete \rscript{readODS}.
\begin{lstlisting}[style=RScript]
# install.packages("readODS")
library(readODS)

datos_ods <- read_ods("datos/mi_calc.ods")

# Especificar hoja
`datos_hoja3` <- read_ods("datos/mi_calc.ods", sheet = 3)
\end{lstlisting}
\end{frame}




\begin{frame}[fragile]{Ficheros .csv}
Los archivos CSV (Comma Separated Values) son el formato más estándar para intercambio de datos. Son archivos de texto plano donde las columnas están separadas por comas. No almacenan nada que no sea dato (a diferencia de un fichero .xlsx que guarda colores, formato de texto, etc.).

Usamos la función \rscript{read_csv()} del paquete \rscript{readr}:
\begin{lstlisting}[style=RScript]
datos_csv <- read_csv("datos/archivo.csv")	
\end{lstlisting}

\titleitem{El problema del punto y coma}
En países de habla hispana (y Europa), usamos la coma \rscript{,} para decimales (ej: 3,14). Por lo tanto, Excel suele guardar los CSV usando punto y coma \rscript{;} como separador de columnas para no confundirse. Si tu archivo usa \rscript{;} como separador, deb\'es usar \rscript{read_csv2()}:
\begin{lstlisting}[style=RScript]
# Para CSV con separador ";" y decimal ","
datos_latino <- read_csv2("datos/archivo_latino.csv")	
\end{lstlisting}

Aunque tambi\'en pod\'es leerlos con \rscript{read_csv()}. Averig\"u\'a c\'omo con \rscript{?read_csv}.
\end{frame}




\begin{frame}[fragile]{Ficheros .tsv}
Los archivos TSV (Tab Separated Values) son similares a los CSV pero usan el caracter tabulador para separar columnas. Son muy comunes en bioinformática o exportaciones de bases de datos.
\begin{lstlisting}[style=RScript]
datos_tsv <- read_tsv("datos/archivo.tsv")	
\end{lstlisting}
\end{frame}



\begin{frame}[fragile]{Ficheros .rds}
El formato \rscript{.rds} es un formato binario propio de R. Sirve para guardar \textbf{un único objeto} de R (un data frame, una lista, un modelo estadístico, etc.) manteniendo todas sus propiedades intactas (tipos de datos, atributos, etc.).

Es muy útil para guardar resultados intermedios que tardaron mucho en calcularse.

\titleitem{Guardar un objeto}
\begin{lstlisting}[style=RScript]
saveRDS(mis_datos, "datos_procesados/mis_datos_limpios.rds")
\end{lstlisting}


\titleitem{Leer un objeto}
Ac\'a debemos asignar el resultado a una variable, ya que el archivo no guarda el nombre del objeto, solo su contenido.
\begin{lstlisting}[style=RScript]
datos_recuperados <- readRDS("datos_procesados/mis_datos_limpios.rds")
\end{lstlisting}
\end{frame}




\begin{frame}[fragile]{Fichero .RData}
El formato \rscript{.RData} (o \rscript{.rda}) también es propio de R, pero permite guardar múltiples objetos a la vez. Es como guardar \quotation{todo el espacio de trabajo} o una parte de él.

\titleitem{Guardar objetos}
\begin{lstlisting}[style=RScript]
save(`objeto1`, `objeto2`, `:ky:file` = "resultados/mi_trabajo.RData")
\end{lstlisting}


\titleitem{Leer objetos}
A diferencia de \rscript{readRDS}, la función \rscript{load()} no se asigna a una variable. Simplemente \quotation{carga} los objetos en tu entorno con los nombres que tenían cuando se guardaron.

\begin{lstlisting}[style=RScript]
load("resultados/mi_trabajo.RData")
#objeto1 y objeto2 aparecerán en tu panel de Environment.	
\end{lstlisting}
\end{frame}




\begin{frame}[fragile]
	Por lo general, prefiere \rscript{.rds} sobre \rscript{.RData} para guardar datos, ya que te da control sobre el nombre de la variable al cargarla (\rscript{datos <- readRDS(...)}). El \rscript{.RData} puede \textbf{sobrescribir} variables existentes sin aviso si tienen el mismo nombre.
\end{frame}





\begin{frame}[fragile]{Gr\'aficas sencillas}
	Aunque \rscript{`ggplot2`} es el estándar de oro para visualización en R (y lo veremos más adelante), R base ofrece funciones gráficas muy potentes para realizar un análisis exploratorio de datos rápido. Cuando recibimos un dataset nuevo, queremos ver la forma de los datos rápidamente, sin preocuparnos por colores bonitos o etiquetas perfectas.
\end{frame}



\begin{frame}[fragile]{plot}
	Es una función genérica que cambia su comportamiento según qué datos le pasemos. 
	\begin{itemize}
		\item Si le pasamos dos vectores numéricos (\rscript{x} e \rscript{y}), hace un gráfico de dispersión (scatterplot).
		\item Si le pasamos un factor, hace un gráfico de barras.
	\end{itemize}
\begin{lstlisting}[style=RScript]
# x = Cilindros, y = Millas por galón
plot(mtcars$cyl, mtcars$mpg)

# Relación entre Potencia y Consumo
plot(x = mtcars$hp, y = mtcars$mpg)
\end{lstlisting}

\begin{frame}[fragile]
	El histograma es fundamental para entender la distribución de una variable numérica continua. Nos dice si los datos están centrados, si son asimétricos o si hay valores inusuales.
\begin{lstlisting}[style=RScript]
# Distribución de la potencia de los autos
hist(mtcars$hp)	
\end{lstlisting}

	Podemos aumentar o disminuir los cortes con el argumento \rscript{breaks} para ver más o menos detalle.
\begin{lstlisting}[style=RScript]
hist(mtcars$hp, breaks = 20)
\end{lstlisting}

\begin{frame}[fragile]{boxplot}
El diagrama de caja y bigotes es la mejor herramienta para visualizar la distribución estadística robusta. Nos muestra la mediana (la línea gruesa), los cuartiles (la caja) y los outliers o valores atípicos.
\begin{lstlisting}[style=RScript]
# Boxplot simple de una variable
boxplot(mtcars$mpg)

# Boxplot comparativo: Rendimiento según cantidad de cilindros
# La fórmula se lee: "Millas por galón en función de Cilindros"
boxplot(mtcars$mpg ~ mtcars$cyl)	
\end{lstlisting}

\begin{frame}[fragile]{barplot}

conteos <- table(mtcars$cyl)

# Luego graficamos
barplot(conteos)
